## 1. 用户回答区（Q&A）

### Q1: 你的 agent harness 主要解决什么任务？

> 选一个最具体的场景，越具体越好。
> 例如："自动整理我读过的资料到飞书"、"帮我搭建多 agent 的代码评审流程"、"在我研究新技术时持续输出周报"。
> 越具体，蓝图越能落地。

**A:**
> 我是想搭建一个属于我自己工作流agent，飞书只是我平时经常用的动作

### Q2: Harness 的"环境"是什么？（你的 agent 在哪里行动、能看到什么、能改变什么）

> Harness Engineering 三大支柱：**action space**（agent 能做什么）/ **observation space**（agent 能看到什么）/ **state**（agent 维护的状态）。
> 例：action = 调 lark-cli + WebFetch + Read/Write 文件；observation = 飞书文档 + 本地 progress.md；state = `_meta/` 三件套。

**A:**
> 我希望我的agent能根据用户的行为去实现自我升级和进化，同时能将成长日志不间断持续的更新到飞书上面去方便我查阅和监测。同时希望能更好的使用元思想添加到里面去，具体的元思想可以参考飞书文档：https://waytoagi.feishu.cn/wiki/KGbewcwM1ic0kFk2A58cL8OanFh 这个文档内容，你读出来理解一下

### Q3: 你的 agent 出错时，你最害怕哪一类失败？

> 例如：
> - 写错文档 / 误删数据 / 调错 API
> - 循环不停 / 超时
> - 产出不符合期望 / 凭空捏造
> - 静默失败（不报错但没做对）
>
> 你最害怕哪 1-2 类？这决定了你最优先建什么 guardrail。

**A:**
> 误删数据 ，产出不符合期望 ，凭空捏造，静默失败。致命型的都害怕

### Q4: 你怎么衡量"agent 做得好"？（eval 标准）

> 没有 eval 就没有迭代。先想最简单的 eval：
> - 用户手动评分（1-5）？
> - 任务成功率（true/false）？
> - 产出长度 / 时间 / 错误数？
> - 多次运行的一致性？
>
> 一开始可以很粗糙，关键是**写下来**。

**A:**
> 自我进化，并且每次执行任务会汇报他的工作情况，正确率高，元的思想合理运用，并且知道何时用普通模式，何时用team模式执行。但是考虑到有的第三发，例如：deepseek，他们是没有team模式，或者是我猜错了，但是一定要准确率高。并且我能实时观察到他在干什么事情，还会自行自己记忆功能

### Q5: 1 个月后（2026-5-28）你希望这个 harness 至少能做到什么？

> 给一个里程碑，避免无限延展。
> 例：能自动处理 10 篇飞书文档 / 跑通 3 个不同 skill / 完成第一次 L2 复盘。

**A:**
> 能完全应用到我得项目里面去，从产品需求整理和产出，到最后项目落地（涉及代码开发，需求整理，前端后端都涉及，多种开发语言都会有）。完全能通过这个harness agent落地

## 4. 优先级（Priority）

### 排序原则

1. **先治理元，后业务元**（B 类先于 A 类）—— 元思想原则："**先选一条最核心的路径，把元、顺序、校验、回滚先做对**"
2. **先 §1.Q3"致命"对应的 guardrail**（B1 进化 + B3 记忆 + A8 风险治理 → 都对应你怕的"凭空捏造 / 静默失败"）
3. **先把既有 skill 的闭环跑通，再扩**——knowledge-curator 还没收过 user_feedback，先跑顺它

### 5 步落地法（元思想模板）

| Step | 动作 | 这个项目里的具体化 |
|------|------|-------------------|
| 1 | 拆元 | §3 已列 16 个元 |
| 2 | 写职责边界 | 每个 skill 的 SKILL.md 顶部强制有 "适用 / 不适用" 表 + 5 特征自检 |
| 3 | 画顺序 + 跳过/插队 | 节奏编排表 C1（M3 做） |
| 4 | 关键牌的触发条件 + 成功指标 | logs/runs.jsonl + user_feedback 是雏形；M2 进化元固化 |
| 5 | 加校验、修复、回滚 | A6 校验元 + A8 风险治理元（v0.2，不在本月） |

## 6. 风险与诚实声明（Risk & Honest Notes）

按 golden rules 必填，每条都是当前 v0.1 已知会出问题的地方。

| # | 风险 | 失败模式 | Mitigation |
|---|------|---------|-----------|
| R1 | Q5 目标"1 个月完成 SDLC 全套"不切实际 | 强行做 → 16 个半成品元，每个都不"成熟"（按元思想 4 个判断问题都不过关） | §4 已降级 M1-M5；**等用户确认接受降级 / 或用户显式拒绝**（待你拍板） |
