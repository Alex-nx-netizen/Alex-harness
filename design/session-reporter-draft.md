## §1 用户决策区（Q1-Q5）

### Q1: 飞书推送目标是什么？

> 当前 lark-cli 可支持的目标：
> a. **飞书文档（Doc）**：追加内容，像日记一样可读；适合人工浏览
> b. **飞书多维表格（Base）**：每条会话一行记录，可筛选/排序/统计；适合数据分析
> c. **飞书 IM（Bot 消息）**：即时推送给自己，看完就算；适合快速感知
> d. **混合**：Base 存结构化数据 + IM 推摘要提醒
>
> 推荐：**d**（Base 做主存储，IM 推 2-3 行摘要提醒；可查 + 即时感知）

### Q3: 推送哪些内容？

> a. **全量 progress.md 段落**：原文推送（含 🔲 / ✅ / 🎯 所有行）
> b. **✅ 完成项 only**：只推带 `✅` 的行，过滤掉 `🔲 / ❌ / ⏭`
> c. **摘要压缩**：让 Claude 把段落压成 2-3 句话再推（需要 LLM 介入）
> d. **结构化字段提取**：提取 task_id / skill / 关键产出，作为 Base 字段
>
> 推荐：**b + d 组合**（✅ 原文 → Base 的 detail 字段；同时提取结构化字段）

### Q4: 增量如何追踪？（最关键的设计问题）

> a. **cursor 文件**：`logs/push-cursor.json` 记录上次推到哪个 run_id；每次只推新的
> b. **frontmatter 标记**：progress.md 里每段打一个 `pushed: true` 标记（改源文件）
> c. **md5 内容哈希**：推送前算哈希，推成功后记到 cursor；内容不变就跳过
> d. **时间戳**：记录上次推送时间，只推比它新的段落
>
> 推荐：**a**（cursor 文件最简单、不污染源文件、可 replay；与 evolution-tracker 的 cursor 同模式）

> a. **手动**：`node run.cjs` 我叫才推

## §2 系统架构（v0.0 草案，待 Q1-Q5 确认）

```
progress.md
    ↓ parse（按 ## 日期 / ### 会话 切分）
新会话列表（与 cursor 对比）
    ↓ filter（只取 ✅ 行 + 结构化提取）
push_items[]
    ↓ lark-cli base +records-create（每条一行）
    ↓ lark-cli im +messages-send（2行摘要）
cursor 更新
```

## §3 5 特征自检

| 特征 | 评估 | 说明 |
|------|------|------|
| 独立 | ✅ | 只读 progress.md + 写 cursor；不修改 progress.md 本身 |
| 足够小 | ✅ | parse + push + cursor 单文件 < 300 行 |
| 边界清晰 | ✅ | 不替代 progress.md；只做镜像推送 |
| 可替换 | ✅ | 推送目标（doc/base/im）可切换，逻辑无硬依赖 |
| 可复用 | ⚠️ | v0.1 特化飞书；v0.2 抽象 output adapter 可接其他平台 |

## §4 失败模式

| 失败 | 场景 | Guardrail |
|------|------|-----------|
| 推送成功但 cursor 未更新 → 重复推 | 写 cursor 前崩溃 | 先写 cursor 再推 OR Base 加 run_id 唯一键去重 |
| progress.md 格式变化 → parse 错误 | 手动改了段落格式 | parse 严格匹配 `## YYYY-M-D` 和 `### ` 前缀；不匹配则跳过不 crash |
| 飞书 API 失败 → 数据丢失 | 网络/token 过期 | retry 2 次；失败后打印错误 + 保留 push_queue.jsonl 待手动重推 |
| cursor 损坏 → 全量重推 | JSON 解析失败 | 读 cursor 出错 → 只推最近 3 条（safe default）不全量 |
