# v0.9 — minimal-mode 默认（基于真实日志的承认）

> **写作时间**：2026-5-14
> **数据来源**：2026-5-8 → 5-13 两个真实项目，35 次 helix --start，24 次 finalize，**100% `phases_run=0`**
> **状态**：v0.9 已定，逐步落地

---

## 1. 为什么改

v0.8.1 把 helix 设计成 "9 phase 必跑硬契约"。真实日志说明：

| 指标 | 数据 |
|---|---|
| 真实 `--start` | 35 次 |
| `--finalize` | 24 次 |
| 中途被 `abort_stale` 打断 | 9 次（25.7%）|
| `phases_run = 0` 的 finalize | **24 / 24 = 100%** |
| LLM 真实调用 `--report` | **0 次** |
| mode-router `score = 0` | **12 / 12 = 100%**（中文正则全 miss）|
| 治理元 script_not_found | 最近 2 次 finalize 才暴露 |

→ **结论**：用户用 helix 时实际工作流是 `--start → 直接干活 → --finalize`。9 phase 是装饰，不是契约。

继续假装"必跑"等于自欺。v0.9 承认事实，把 phase 从"必跑"降为"opt-in"。

## 2. 设计原则

1. **数据驱动**：以日志为唯一真相源，不基于"我以为用户会喜欢"
2. **不删代码**：phase 脚本全部保留，只下移它们的"必跑性"
3. **加新 flag 而非破坏旧 API**：`--start` 行为不变，引入 `--deep` 表达"我要 9 phase"
4. **失败软化**：phases_run=0 不再 promise=NOT_COMPLETE，改成 trivial_ok
5. **stale run 自动收尾**：30 分钟没活动 → 自动 abandoned 归档

## 3. 改动清单

### 3.1 helix（核心）

| 行为 | v0.8.1 | v0.9 |
|---|---|---|
| `--start <task>` 默认模式 | full 9-phase | **minimal**（只 start + finalize）|
| `--start --deep <task>` | n/a | full 9-phase（旧行为）|
| `--start` 撞到旧 current-run | error 阻塞 | **>30min 自动 abandoned 归档**，启新的 |
| `--finalize` 时 `phases_run=0` | promise=NOT_COMPLETE | mode=minimal → **promise=COMPLETE_MINIMAL**（不卡）|
| 治理元 script 找不到 | 默默 skip（最近才补 log）| 同样 skip，但显式记录到 `governance_auto_results` |

### 3.2 mode-router（修匹配，不砍）

- 加中文 backend / API 关键词：`接口` / `API` / `服务端` / `controller` / `entity` / `repository`
- 加中文 frontend 关键词：`界面` / `画面` / `页` / `表单` / `按钮` / `widget`
- 加单元测试：用外部业务项目真实任务里的 5 条 task_desc 跑，断言 score > 0

> 仍保留 mode-router 的存在 —— 真实跑通后再决定要不要进一步砍。

### 3.3 a2-repo-sensor（清残留）
