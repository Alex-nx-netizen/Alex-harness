## §0 一句话责任

**根据任务复杂度 + 当前 LLM 能力，决定走 solo（单会话）还是 team（多 agent 并行），并输出可执行的路由建议。**

## §1 用户决策区（Q1-Q5）

### Q1: 你日常什么情况下觉得"一个 agent 不够用"？

> 例：
> - 任务太长，一个会话跑不完（context 不够）
> - 任务可以并行拆分（如同时做前端 + 后端）
> - 需要独立审查（一个写代码 + 另一个 review）
> - 需要不同专业视角（产品 + 技术 + 测试）
>
> 你最常遇到哪 1-2 类？

**A: 并行拆分（同时做前端+后端）+ 独立审查（写+review 分开）**

### Q2: 你希望 mode-router 的触发方式是什么？

> a. 自动判断 / b. 我显式说 / **c. 推荐+确认** / d. 混合

**A: C — 推荐+确认**（router 任何时候都先给推荐，等我说确认才算数）

### Q3: "team 模式" 在你这里意味着什么？

> a. subagent 并行 / b. sequential / **c. peer review** / d. specialized roles
>
> 推荐：先只做 **a + c**

**A: 推荐 = a (subagent 并行) + c (peer review)**，b/d 放 v0.2

### Q4: 第三方 LLM（deepseek 等）时？

> **a. 自动降级** / b. 提示切换 / c. 忽略

**A: A — 自动降级 solo + ⚠️ 提示**

### Q5: mode-router 的输出格式？

> **a. 打印 + b. 写日志** / c. 注入 prompt / d. 只返回变量

**A: 推荐 = a (终端打印) + b (`_meta/mode-router-log.jsonl`)**

## §2 路由决策矩阵（v0.1，Q1-Q5 锁定后）

> Q2=C 核心规则：**任何路由都先推荐，等确认**；仅两个例外：显式指令直接执行、非 Claude 强制降级

| 信号 | 判断 | 路由结果 |
|------|------|---------|
| 任务含并行信号（并行/前端+后端/同时做/拆分） | 高并行性 | **推荐 team/subagent，等确认** |
| 任务含审查信号（review/审查/一写一审） | 审查需求 | **推荐 team/peer_review，等确认** |
| 无明确信号 | 默认 | **推荐 solo，等确认** |
| 用户说"solo" / 任务含 solo 关键词 | 显式指令 | **solo（直接执行，跳过推荐）** |
| 用户说"team" / 任务含 team 关键词 | 显式指令 | **team（直接执行，跳过推荐）** |
| 当前 LLM ≠ Claude | 能力限制 | **强制 solo + ⚠️ 提示（不等确认）** |

## §3 5 特征自检

| 特征 | 评估 | 说明 |
|------|------|------|
| 独立 | ✅ | 只读任务描述 + LLM 信息，输出路由建议，不执行任务 |
| 足够小 | ✅ | 决策矩阵 + log 写入，单文件 executor < 200 行 |
| 边界清晰 | ✅ | 不执行 team，只决定是否 team；不修改其他 skill |
| 可替换 | ✅ | 输出格式标准化（solo/team + reason），下游无硬依赖 |
| 可复用 | ⚠️ | v0.1 硬编码决策矩阵；v0.2 参数化 |

## §4 失败模式

| 失败 | 场景 | Guardrail |
|------|------|-----------|
| 误判复杂度 → solo 做了应 team 的任务 | 检测规则不足 | 允许用户覆盖 + 记录反例到 log（feedback loop） |
| team 模式派出 subagent 但 LLM 不支持 | deepseek fallback 未触发 | 启动时检测 `process.env.ANTHROPIC_MODEL` 或 API 响应判断 |
| 推荐太多打扰流程 | Q2=C 是用户选的，可接受 | 显式 solo/team 仍可跳过推荐 |
