## P-2026-4-30-002 skill_feedback CLI 加 fix_notes 质量门 + 关键词宽容化

**direction**: `feedback_quality_gate`
**source**: `meta_loop_observation`（evolution-tracker 在 context-curator 上首跑产出 0 议案 → 暴露此问题）
**evidence_runs**: `["evt-2026-4-30-context-curator-144737"]`
**evidence_findings**: `["F-017"]`

**What**:
2 处修改（A + B 二选一或都做）：

**A. `_meta/skill_feedback.cjs` 加质量门**
- fix_notes 长度 < 30 字符 → warn "建议 ≥ 30 字 + 含具象词（如失败/缺/没/Phase/应该/具体哪节）"，给 `--allow-short` 才放行
- 且默认要求 fix_notes 非空（"..."、空字符串、纯标点拒绝）

**B. `.claude/skills/evolution-tracker/SKILL.md` 扩展 SIGNAL_KEYWORDS**
- 当前：`[不会, 不能, 没改, 没变, 静默, 凭空, 捏造, 失败, 进化, 自我, 缺, Phase, 应该, fix]`
- 加：`[压成, 看不到, 激进, 不到, 太激, 太多, 截断, 假阳性, 噪声, 漏, 重复, 想要, 希望]`
- 让具象描述也能命中

**Why**:
F-017 实证：Alex 4-30 给 context-curator 打分用了 "..." → evolution-tracker 跑出 0 议案 → meta-loop 看似闭合但其实空转。这暴露了 skill_feedback CLI 假设"用户会写好 fix_notes"，实践中**用户会偷懒**。如果不在 CLI 层制度化，每次都靠用户记得写 = 不可持续。

**Where**:
- A: `_meta/skill_feedback.cjs` rateRun() 函数加 fix_notes 校验（参考已有 rating 1-5 校验的模式）
- B: `.claude/skills/evolution-tracker/SKILL.md` frontmatter SIGNAL_KEYWORDS 字段

**Risk**:
1. **A 太严打断流**：用户急着打分时被门拦住 → mitigate: `--allow-short` 转义 + warn 信息明示这条 run 在 evolution-tracker 里会变 uncategorized
2. **B 关键词扩太宽误命中**：把无关词当信号 → mitigate: 先小步加 5-8 个明确具象词；评估 1-2 周 false positive；再决定是否再扩
3. **A + B 同时做的 over-engineering**：可能只需一个就够 → mitigate: 推荐先做 A（卡输入端），B 看 1-2 周数据再决定
4. **覆盖现有 user_feedback**：A 的拒绝不能误伤已写过的有内容评分 → mitigate: 仅在写入路径校验，--list 不影响
